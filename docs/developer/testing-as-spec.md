# Тестирование как спецификация

> Для разработчиков, пишущих тесты для кода IWE (MCP-сервисы, обработчики бота, скрипты). Ссылка из [developer-guide.md](developer-guide.md) на переходе между станцией 3 (Проектирование) и станцией 4 (Работа).

## Принцип

Тест описывает поведение системы на её **границе** — что она принимает и что возвращает, а не то, как она устроена внутри.

**Порядок:** сначала пишется тест (спецификация), потом код, который его проходит.

**Контракт:** любая реализация, проходящая все тесты границы, приемлема — независимо от внутреннего устройства.

## Работа с Claude

- Просить Claude сначала описать тесты, потом писать код.
- Если Claude не понимает границу — пусть спрашивает, не угадывает.
- Стоп-сигнал: если тест ломается при равноценной замене реализации (тот же вход/выход, другой способ вычисления) — значит, тест проверяет деталь реализации, а не контракт. Переписать тест на языке границы.

## Что тестировать на границе (пример — MCP-сервис)

- Список инструментов соответствует ожидаемому — нет лишних, нет пропущенных.
- Авторизация принимает и отклоняет корректно.
- Формат ответа соответствует заявленному контракту.
- Каждый инструмент возвращает ровно заявленные данные, не больше.
